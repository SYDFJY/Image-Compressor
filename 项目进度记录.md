### 技术决策

1. **Maven Wrapper 为什么用 3.9.6？** — 3.9.x 最后支持 JDK 8 的版本线，Maven 4.0+ 需要 JDK 17+
2. **pom.xml 中用 `${property}` 管理版本号** — 统一版本控制，后续升级只需改一处
3. **assembly plugin 配置 `appendAssemblyId=false`** — 让 fat JAR 输出为 `image-compressor-1.0.0.jar` 而非 `image-compressor-1.0.0-jar-with-dependencies.jar`

| 编译验证 | ✅ 通过 | `mvn compile` BUILD SUCCESS，12 个源文件 0 错误 0 警告 |

### 技术决策

1. **FileUtil 的防抖生成唯一文件名** — 支持最多 9999 个递增编号，超出后改用时间戳避免死循环
2. **ImageUtil 缩略图/预览图分离** — 缩略图 (`generateThumbnail`) 用于列表，固定 200×150；预览图 (`generatePreview`) 用于对比面板，最大 1024×768；不同场景不同尺寸避免内存浪费
3. **ValidateUtil 返回 ValidationResult 而非 boolean** — 携带错误消息字符串，方便 UI 层直接显示给用户，无需额外翻译步骤
4. **ConfigService 防抖机制** — 滑块拖拽可能每秒触发几十次 saveConfig 调用，用 500ms Timer 防抖合并为一次磁盘写入；`shutdown()` 强制刷盘保证退出时不丢数据
5. **损坏配置文件自动备份** — 读取到损坏 JSON 时自动重命名为 `.bak`，然后写入新的默认配置，防止用户手动修改出错后数据永久丢失

## 2026-07-09 · Bug 修复轮次 3：启动画面卡死

### 问题发现

用户反馈：启动时出现蓝色弹窗（SplashScreen），无法关闭，只能强制结束进程。

### 根因分析

`App.java` 中原代码：

```java
splash.setVisible(true);
// ... 初始化 ...
Thread.sleep(300);   // ← EDT 上调用 sleep！
splash.dispose();
```

**致命问题**：`Thread.sleep(300)` 在 EDT（事件调度线程）上执行。Swing 是单线程模型，EDT 被阻塞时无法响应任何事件——包括 `dispose()` 调用和窗口关闭事件。如果 `sleep()` 前抛出任何异常，`dispose()` 永远不会执行，JWindow 永远挂在屏幕上。

### 修复方案（三层保护）

| 层 | 机制 | 作用 |
|----|------|------|
| 1 | `javax.swing.Timer` 替代 `Thread.sleep()` | Timer 在 EDT 上注册回调但不阻塞，300ms 后自动 `dispose()` |
| 2 | `try-finally` 包裹全部初始化 | 任何异常都会在 finally 中 `splash.dispose()` |
| 3 | 点击启动画面手动关闭 | `mouseClicked` → `dispose()`，用户可手动关闭 |

### 涉及文件

| 文件 | 改动 |
|------|------|
| `App.java` | Timer 替代 sleep + try-finally 保护 |
| `SplashScreen.java` | 新增 click-to-close 兜底监听器 |

## 问题日志（长期）

> 记录开发全程重大问题和对应解决方案，作为实训报告素材。

### #1 Thumbnailator 不调用 scale/size 导致压缩静默失败

**现象**：格式转换时抛出 `"size is not set"` 异常。

**原因**：Thumbnailator 0.4.20 的 `Builder` 在调用 `toFile()` 前必须至少调用一次 `scale()` 或 `size()`，即使不需要缩放也要显式 `scale(1.0)`。`ScaleMode.NONE` 原实现为空（no-op），不满足要求。

**解决**：`applyScale()` 的 NONE/default 分支改为 `builder.scale(1.0)`。

**教训**：第三方库的"默认行为"不可靠，关键路径必须有测试覆盖。

---

### #2 Windows 文件锁定导致 Thumbnailator 二次读取失败

**现象**：`ImageIO.read()` 读取尺寸后，`Thumbnails.of()` 再次打开同一文件时抛出 `IOException`。

**原因**：Windows 上 `ImageIO.read()` 全量解码大图后可能未释放文件句柄（取决于 ImageReader 实现），导致后续 `Thumbnails.of()` 无法打开文件。

**解决**：用 `ImageReader.setInput()` 仅读 SOF 元数据获取尺寸（~2KB），替代 `ImageIO.read()` 全量解码（~72MB）。同时解决了 CMYK JPEG 解码失败问题。

**教训**：Windows 文件锁比 Linux 严格得多，跨平台开发时必须注意文件句柄管理。

---

### #3 CMYK 色彩空间 JPEG 无法读取

**现象**：部分照片（尤其是单反相机直出）导入后尺寸显示为 0×0，压缩失败。

**原因**：JDK 8 的 JPEG ImageReader 不支持 CMYK 色彩空间，`ImageIO.read()` 返回 null。这些照片通常来自专业相机或 Photoshop 导出。

**解决**：改用 `ImageReader.setInput()` 读取 SOF 标记中的尺寸（标记段不涉及色彩空间解码），并在零尺寸时给出明确提示："CMYK 色彩空间 JPEG，建议用画图/Photoshop 另存为标准 RGB 格式"。

**教训**：JPEG 格式有多种色彩空间变体，不能假设所有 JPEG 都是 RGB。

---

### #4 JDK 8 不支持 WebP 格式

**现象**：输出格式中选择 WebP 后压缩失败。

**原因**：WebP 由 Google 在 2010 年发布，JDK 8（2014 年）的 ImageIO 不包含 WebP Reader/Writer。JDK 原生 WebP 支持需要 JDK 15+。

**解决**：从所有枚举、下拉框、文件过滤器中移除 WebP 选项。如需支持可通过 TwelveMonkeys 插件扩展，但超出实训范围。

**教训**：JDK 8 的 ImageIO 格式支持很有限（Reader: JPG/PNG/BMP/GIF/WBMP；Writer: JPG/PNG/BMP/GIF/WBMP），引入新格式前必须验证 JDK 8 兼容性。

### #5 EDT 阻塞导致启动画面卡死

**现象**：蓝色 SplashScreen 弹窗无法关闭。

**原因**：`Thread.sleep(300)` 在 EDT 上执行。Swing 是单线程事件循环，sleep 期间无法处理任何 UI 事件（包括窗口关闭）。

**解决**：三重保护——
1. `javax.swing.Timer` 替代 sleep（不阻塞 EDT）
2. `try-finally` 确保异常时也能 dispose
3. 点击画面手动关闭兜底

**教训**：**永远不要在 EDT 上 sleep/wait/执行耗时操作**。任何延迟需求都用 `javax.swing.Timer` 或 `SwingWorker`。

### #8 VideoPreviewPanel 初始化顺序导致 NPE 崩溃

**现象**：应用启动时在 EDT 中抛出 NullPointerException，堆栈指向 `VideoPreviewPanel.createInfoPanel(160)` → `Container.addImpl`，应用无法启动。

**原因**：`VideoPreviewPanel` 构造函数中执行顺序为：
```java
emptyPanel = createEmptyPanel();
infoPanel = createInfoPanel();          // ← 内部引用 this.comparisonPanel
comparisonPanel = createComparisonPanel();  // ← 字段的初始化在 infoPanel 之后
```
`createInfoPanel()` 内部执行 `panel.add(comparisonPanel, BorderLayout.SOUTH)`，但此时 `comparisonPanel` 字段尚未赋值（为 `null`），导致 `Container.addImpl()` 抛出 NPE。编译器无法检测此类初始化顺序问题——字段默认为 `null`，只有运行时才暴露。

**解决**：将 `comparisonPanel = createComparisonPanel()` 移到 `createEmptyPanel()` 和 `createInfoPanel()` 之前。

**涉及文件**：`view/VideoPreviewPanel.java`（+3/-1）

**教训**：构造函数中调用实例方法时，如果这些方法依赖其他实例字段，必须确保字段的赋值在这些方法调用之前完成。编译器对这类初始化顺序问题零检测，只能靠人工审查或运行时测试发现。

### #9 EDT 阻塞导致视频模式切换和播放卡顿/无响应

**现象**：
1. 点击播放按钮后 UI 冻结 2+ 秒，ffplay 窗口才弹出
2. 首次切换到视频模式时界面无响应 ~10 秒
3. 视频压缩失败率极高，且错误信息仅显示 "非零退出码: X"，无具体原因

**原因**（三层问题）：
1. **播放 EDT 阻塞**：`killFfplayProcess()` 含 `waitFor(2, SECONDS)` 同步等待，直接写在按钮 ActionListener 中执行，阻塞 EDT
2. **模式切换 EDT 阻塞**：`checkFfmpegAvailable()` 首次调用时启动 2 个子进程（ffmpeg + ffprobe），各等 5 秒超时判断，全在 EDT 上执行
3. **压缩错误信息被丢弃**：`executeCompress()` 只解析 stderr 进度行，FFmpeg 报错行全被丢弃；stdout 缓冲区满导致 FFmpeg 管道死锁

**解决**：
| 问题 | 修复 | 涉及文件 |
|------|------|---------|
| 播放阻塞 | 所有阻塞操作（checkFfplayAvailable / killFfplayProcess / playVideo / waitFor）移入 "FFplay-Thread" 后台线程 | `controller/MainController.java` |
| 模式切换阻塞 | 启动时 "FFmpeg-Detection-Thread" 后台预检测 FFmpeg/FFplay，缓存热后 EDT 调用瞬间返回 | `controller/MainController.java` |
| 压缩无错误信息 | stderr 循环保留最后 5 行非进度输出拼入异常消息；新增 stdout 排空线程防死锁 | `service/VideoCompressUtil.java` |

**涉及文件**：`controller/MainController.java`（~40行）、`service/VideoCompressUtil.java`（~15行）

**教训**：
1. **任何可能阻塞的调用都不能出现在 EDT 上** — `Process.waitFor(timeout)` 看似有超时就没事，但它同步阻塞调用线程，EDT 上每一毫秒都是致命
2. **子进程 stderr 可能包含救命信息** — 只解析格式已知的行（进度），把其余行全丢弃，等于蒙着眼睛调试。Ring buffer 是最小代价保留诊断信息的方法
3. **管道缓冲区满是一个隐匿杀手** — FFmpeg stdout 如果没人读，管道缓存写满后 FFmpeg 阻塞在 `write()`，看起来像"卡死"。stdout 即使不用也得排空

### #10 同一视频多次压缩时输出文件冲突

**现象**：第一次压缩成功（如 `video.mp4` → `video_compressed.mp4`），第二次压缩同一视频直接失败，提示「输出文件已存在，请启用覆盖或修改命名规则」。

**原因**：`VideoCompressConfig.overwrite` 默认值为 `false`，`VideoCompressService.compress()` 在启动 FFmpeg 之前做预检：输出文件存在 + overwrite=false → 直接返回失败。即使 FFmpeg 命令中有 `-y` 标志也没机会执行。此外 `VideoParamPanel` 复选框标签写的是「覆盖原文件」，但视频模式**永远不会覆盖原文件**（总有 `_compressed` 后缀），标签有误导性。

**解决**：
| 改动 | 说明 |

### #11 视频缩放后编码失败（force_original_aspect_ratio → 奇数宽高）

**现象**：将分辨率从「原始」改为任意其他值（480p/720p/1080p/4K）后，压缩失败。

**原因**：`scale=W:H:force_original_aspect_ratio=decrease` 在竖屏视频（如手机拍摄的 1080×1920）上，自动计算的宽度可能为奇数（如 405px）。H.264/VP9 的 yuv420p 像素格式要求宽高均为偶数，奇数 → FFmpeg 编码失败，报 "width not divisible by 2"。

**解决**：将 scale 滤镜改为 `scale=W:-2`。`-2` 是 FFmpeg 内置占位符，含义为「按宽高比自动计算该维度，并向下取偶数」（即 `trunc(oh*a/2)*2`）。目标分辨率宽度（854/1280/1920/3840）本身均为偶数，无需额外处理。

**涉及文件**：`service/VideoCompressUtil.java`（+5/-3）

**教训**：
1. **视频编码对尺寸有硬约束** — yuv420p 要求宽高为偶数（色度子采样 4:2:0 导致 UV 平面 = 亮度平面的一半）。任何自动计算的维度都必须保证满足此约束
2. **FFmpeg 的 `-2` 比手动表达式更可靠** — 比自己写 `trunc(oh*a/2)*2` 简洁且不易出错
3. **测试必须覆盖竖屏视频** — 手机拍摄的竖屏视频（9:16）是常见场景，开发时容易只测横屏（16:9）而漏掉此 case

### #12 视频播放无响应 — ffplay stdout 管道死锁

**现象**：点击 ▶ 播放按钮后应用无响应，ffplay 窗口不弹出。之前已修复两次 EDT 阻塞问题（#9），但播放仍然卡死。

**原因**：`VideoUtil.playVideo()` 启动 ffplay 子进程后从不读取 stdout。ffplay 持续向 stdout 输出初始化/SDL 调试信息，当 OS 管道缓冲区（4-64KB）写满后，ffplay 的 `write()` 系统调用阻塞 → 进程挂起 → Java 端 `waitFor()` 永远等不到返回 → `"FFplay-Thread"` 后台线程永久死锁。

根因与 issue #9 中修过的 `VideoCompressUtil.executeCompress()` stdout 死锁完全一致（commit `06d1741`），当时只修了压缩端，漏了播放端。

**解决**：在 `playVideo()` 中添加 `FFplay-Stdout-Drainer` 守护线程，持续排空进程 stdout。`setDaemon(true)` 确保不阻止 JVM 退出。

**涉及文件**：`util/VideoUtil.java`（+21/-1）

**教训**：
1. **子进程管道死锁是系统性风险** — 任何 `ProcessBuilder.start()` + `waitFor()` 的组合，如果没有读取 stdout/stderr，都可能在输出量超过管道缓冲时死锁。修了一处，就要排查所有同类代码
2. **守护线程是最佳排空方式（JDK 8 下）** — JDK 9+ 的 `Redirect.DISCARD` 更优雅，但 JDK 8 约束下 daemon thread 是最简洁方案：不阻止退出、不占用资源、自动回收
3. **同一类 bug 反复出现说明缺少通用抽象** — 如果以后还要启动更多 ffmpeg 子进程，应该把管道排空逻辑提取为 `ProcessUtil.drainAsync(Process)` 公共方法

## 问题日志（长期 · 续）

### #8 VLC 捆绑版无音频/无控件

**现象**：捆绑版 VLC 被检测到，内嵌播放器能显示画面，但无音频输出、无进度条、无音量控制滑块。

**原因**：VLC 启动时找不到 `plugins/` 目录。仅设置 JNA 的 `NativeLibrary.addSearchPath()` 加载 `libvlc.dll` 不够 — libvlc 启动后还需要加载 `plugins/` 下的音频解码器、视频滤镜等模块。VLCJ 3.x 需要通过 `--plugin-path` 参数告知 libvlc 插件目录位置。

**解决**：三层防护 —
1. Java 层：`EmbeddedMediaPlayerComponent.onGetMediaPlayerFactoryArgs()` 传入 `--plugin-path`
2. 启动脚本层：设置 `VLC_PLUGIN_PATH` 原生环境变量
3. 自动探测层：`VlcUtil.findBundledVlc()` 自动发现捆绑版 VLC 并缓存插件路径

**涉及文件**：`VlcUtil.java`、`VideoPlayerPanel.java`、`dist/启动.bat`、`dist/启动.vbs`

**教训**：JNA 库加载（`NativeLibrary`）只解决了 Java↔Native 的函数调用桥梁，但 VLC 是一个模块化架构 — `libvlc.dll` 只是入口，实际解码/渲染由 `plugins/` 下的 200+ 个模块完成。只加载入口 DLL 不告诉它插件在哪，等于给了钥匙但没告诉门在哪。

### #9 视频单文件压缩重名直接失败

**现象**：单文件压缩时，若输出目录已有同名文件，非覆盖模式下直接返回失败，不会像图片压缩那样自动编号。

**原因**：`VideoCompressService` 是新写的服务类，覆盖检测逻辑来自 `VideoCompressUtil.generateOutputFile()`，但缺少 `FileUtil.generateUniqueFilename()` 自动编号调用。图片压缩的 `CompressService` 早已包含此逻辑，视频服务开发时遗漏了对齐。

**解决**：在 `VideoCompressService.compress()` 中添加与图片压缩一致的非覆盖自动编号逻辑。

**涉及文件**：`VideoCompressService.java`

**教训**：相似功能的服务类应保持行为一致。视频压缩服务开发时参考图片压缩服务的已有逻辑可避免此类遗漏。后续可考虑抽取公共的「文件输出策略」模块。

### #10 批量导出按钮显示"0 个版本"

**现象**：已添加多个导出变体，但压缩按钮始终显示"批量导出 0 个版本"。

**原因**：批量复选框的 `ActionListener` 中硬编码 `updateCompressButtonText(0)`，无论实际变体数量多少都传 0。

**解决**：移除硬编码调用；新增 `onVariantChanged` 回调机制，变体增删时自动触发按钮文案更新，传入实际变体数量。

**涉及文件**：`VideoParamPanel.java`、`MainController.java`

**教训**：硬编码的占位值容易在后续开发中被遗忘。初期可以用硬编码快速跑通流程，但必须在 TODO 注释中标记，并在功能完成后清理。

### #11 视频元数据全部显示"未知" — FFmpeg 未安装/未被发现

**现象**：导入视频文件后，视频信息 Tab 中时长、分辨率、编码、帧率、比特率全部显示"未知"，仅文件名和大小正常。

**原因**：分两层 —
1. **环境层**：系统未安装 FFmpeg（全盘搜索无 `ffprobe.exe`）。Java 没有内置视频元数据解析能力，必须依赖 ffprobe 外部工具。
2. **代码层**：`VideoUtil.resolveCommand()` 仅检查系统属性 `ffmpeg.bin.path` 和系统 PATH。若 ffprobe 由 IDE 安装（如 TRAE SOLO CN 放入其 bin 目录），从 IDE 内启动的 App 能找到（继承 IDE 的 PATH），但从命令行 `mvn exec:java` 启动的 App 找不到（继承系统 PATH）。

**解决**：
1. `VideoUtil.java` — 新增 `autoDiscoverFfmpegPath()` 方法，扫描 10+ 常见 Windows 安装目录（`FFMPEG_HOME`、winget、Chocolatey、Scoop、`C:\ffmpeg\` 等），结果缓存供 `resolveCommand()` 使用
2. `VideoPreviewPanel.java` — 元数据全空且 FFmpeg 不可用时，显示橙色警告条：「FFmpeg 未安装，无法读取视频元数据」+ 安装指引
3. 推荐用户运行 `winget install ffmpeg` 一键安装

**涉及文件**：`VideoUtil.java`、`VideoPreviewPanel.java`

**教训**：依赖外部 CLI 工具的 Java 应用，**不能假设工具一定在 PATH 上**。`ProcessBuilder` 继承 JVM 进程的 PATH，而 JVM 继承启动它的 shell/IDE 的 PATH — 这导致同一份代码在不同启动方式下行为不一致。最佳实践：(1) 文件系统级别的自动发现（遍历常见目录），(2) 允许用户通过系统属性/配置文件指定工具路径，(3) 不可用时给出清晰的安装指引而非静默失败。
