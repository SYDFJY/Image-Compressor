### 技术决策

1. **Maven Wrapper 为什么用 3.9.6？** — 3.9.x 最后支持 JDK 8 的版本线，Maven 4.0+ 需要 JDK 17+
2. **pom.xml 中用 `${property}` 管理版本号** — 统一版本控制，后续升级只需改一处
3. **assembly plugin 配置 `appendAssemblyId=false`** — 让 fat JAR 输出为 `image-compressor-1.0.0.jar` 而非 `image-compressor-1.0.0-jar-with-dependencies.jar`

| 编译验证 | ✅ 通过 | `mvn compile` BUILD SUCCESS，12 个源文件 0 错误 0 警告 |

### 技术决策

1. **FileUtil 的防抖生成唯一文件名** — 支持最多 9999 个递增编号，超出后改用时间戳避免死循环
2. **ImageUtil 缩略图/预览图分离** — 缩略图 (`generateThumbnail`) 用于列表，固定 200×150；预览图 (`generatePreview`) 用于对比面板，最大 1024×768；不同场景不同尺寸避免内存浪费
3. **ValidateUtil 返回 ValidationResult 而非 boolean** — 携带错误消息字符串，方便 UI 层直接显示给用户，无需额外翻译步骤
4. **ConfigService 防抖机制** — 滑块拖拽可能每秒触发几十次 saveConfig 调用，用 500ms Timer 防抖合并为一次磁盘写入；`shutdown()` 强制刷盘保证退出时不丢数据
5. **损坏配置文件自动备份** — 读取到损坏 JSON 时自动重命名为 `.bak`，然后写入新的默认配置，防止用户手动修改出错后数据永久丢失

## 2026-07-09 · Bug 修复轮次 3：启动画面卡死

### 问题发现

用户反馈：启动时出现蓝色弹窗（SplashScreen），无法关闭，只能强制结束进程。

### 根因分析

`App.java` 中原代码：

```java
splash.setVisible(true);
// ... 初始化 ...
Thread.sleep(300);   // ← EDT 上调用 sleep！
splash.dispose();
```

**致命问题**：`Thread.sleep(300)` 在 EDT（事件调度线程）上执行。Swing 是单线程模型，EDT 被阻塞时无法响应任何事件——包括 `dispose()` 调用和窗口关闭事件。如果 `sleep()` 前抛出任何异常，`dispose()` 永远不会执行，JWindow 永远挂在屏幕上。

### 修复方案（三层保护）

| 层 | 机制 | 作用 |
|----|------|------|
| 1 | `javax.swing.Timer` 替代 `Thread.sleep()` | Timer 在 EDT 上注册回调但不阻塞，300ms 后自动 `dispose()` |
| 2 | `try-finally` 包裹全部初始化 | 任何异常都会在 finally 中 `splash.dispose()` |
| 3 | 点击启动画面手动关闭 | `mouseClicked` → `dispose()`，用户可手动关闭 |

### 涉及文件

| 文件 | 改动 |
|------|------|
| `App.java` | Timer 替代 sleep + try-finally 保护 |
| `SplashScreen.java` | 新增 click-to-close 兜底监听器 |

## 问题日志（长期）

> 记录开发全程重大问题和对应解决方案，作为实训报告素材。

### #1 Thumbnailator 不调用 scale/size 导致压缩静默失败

**现象**：格式转换时抛出 `"size is not set"` 异常。

**原因**：Thumbnailator 0.4.20 的 `Builder` 在调用 `toFile()` 前必须至少调用一次 `scale()` 或 `size()`，即使不需要缩放也要显式 `scale(1.0)`。`ScaleMode.NONE` 原实现为空（no-op），不满足要求。

**解决**：`applyScale()` 的 NONE/default 分支改为 `builder.scale(1.0)`。

**教训**：第三方库的"默认行为"不可靠，关键路径必须有测试覆盖。

---

### #2 Windows 文件锁定导致 Thumbnailator 二次读取失败

**现象**：`ImageIO.read()` 读取尺寸后，`Thumbnails.of()` 再次打开同一文件时抛出 `IOException`。

**原因**：Windows 上 `ImageIO.read()` 全量解码大图后可能未释放文件句柄（取决于 ImageReader 实现），导致后续 `Thumbnails.of()` 无法打开文件。

**解决**：用 `ImageReader.setInput()` 仅读 SOF 元数据获取尺寸（~2KB），替代 `ImageIO.read()` 全量解码（~72MB）。同时解决了 CMYK JPEG 解码失败问题。

**教训**：Windows 文件锁比 Linux 严格得多，跨平台开发时必须注意文件句柄管理。

---

### #3 CMYK 色彩空间 JPEG 无法读取

**现象**：部分照片（尤其是单反相机直出）导入后尺寸显示为 0×0，压缩失败。

**原因**：JDK 8 的 JPEG ImageReader 不支持 CMYK 色彩空间，`ImageIO.read()` 返回 null。这些照片通常来自专业相机或 Photoshop 导出。

**解决**：改用 `ImageReader.setInput()` 读取 SOF 标记中的尺寸（标记段不涉及色彩空间解码），并在零尺寸时给出明确提示："CMYK 色彩空间 JPEG，建议用画图/Photoshop 另存为标准 RGB 格式"。

**教训**：JPEG 格式有多种色彩空间变体，不能假设所有 JPEG 都是 RGB。

---

### #4 JDK 8 不支持 WebP 格式

**现象**：输出格式中选择 WebP 后压缩失败。

**原因**：WebP 由 Google 在 2010 年发布，JDK 8（2014 年）的 ImageIO 不包含 WebP Reader/Writer。JDK 原生 WebP 支持需要 JDK 15+。

**解决**：从所有枚举、下拉框、文件过滤器中移除 WebP 选项。如需支持可通过 TwelveMonkeys 插件扩展，但超出实训范围。

**教训**：JDK 8 的 ImageIO 格式支持很有限（Reader: JPG/PNG/BMP/GIF/WBMP；Writer: JPG/PNG/BMP/GIF/WBMP），引入新格式前必须验证 JDK 8 兼容性。

---

### #5 EDT 阻塞导致启动画面卡死

**现象**：蓝色 SplashScreen 弹窗无法关闭。

**原因**：`Thread.sleep(300)` 在 EDT 上执行。Swing 是单线程事件循环，sleep 期间无法处理任何 UI 事件（包括窗口关闭）。

**解决**：三重保护——
1. `javax.swing.Timer` 替代 sleep（不阻塞 EDT）
2. `try-finally` 确保异常时也能 dispose
3. 点击画面手动关闭兜底

**教训**：**永远不要在 EDT 上 sleep/wait/执行耗时操作**。任何延迟需求都用 `javax.swing.Timer` 或 `SwingWorker`。

JDK 8 自带 javapackager 可创建独立 EXE（含捆绑 JRE），但体积达 190MB（剪裁后 114MB），不适合微信/QQ 分享。

**最终方案：原生 C 启动器**
- 用 C + Win32 API 写了一个 130 行的轻量启动器
- 自动在同目录查找 `image-compressor-*.jar`
- 按 3 层策略查找 Java：PATH → 注册表 → 常见安装路径
- 无 Java 时弹出中文错误提示对话框
- 编译后仅 **33KB**

### 问题日志

### #6 Launch4j Maven 插件不可用

**现象**：`mvn package` 报 `com.akathist.maven.plugins:launch4j-maven-plugin:jar:2.5.1 was not found`

**原因**：该插件坐标在 Maven Central 上不存在，且当前网络不可达无法验证。Launch4j 有多个社区 Maven 插件，坐标各异且维护不稳定。

**解决**：放弃 Maven 插件路线，改用 JDK 8 自带的 javapackager + 自研 C 原生启动器。javapackager 生成完整独立包（114MB），C 启动器提供轻量方案（33KB）。

**教训**：依赖外部 Maven 插件前应先验证其在 Central 上的可用性。JDK 8 自带 javapackager 是更可靠的备选方案。原生 C 启动器虽需编译环境，但零依赖、极致轻量。

### #7 javapackager 原生 EXE 双击无反应

**现象**：`javapackager -native image` 生成的 `ImageCompressor.exe` 在命令行启动正常（日志输出"启动完成"），但 Windows 资源管理器中双击毫无反应，不弹窗口也不报错。

**原因**：javapackager 生成的 EXE 并非独立可执行文件 — 它是通过 `packager.dll` 直接加载 JVM 的瘦启动器（17KB）。该启动器依赖 `$APPDIR` 变量解析 runtime 路径，在部分 Windows 10/11 环境下双击时路径解析失败，导致 JVM 加载失败后静默退出。

**解决**：彻底放弃 javapackager 原生 EXE，改用 **捆绑 JRE + VBS 启动脚本** 方案：
- 从 JDK 8 复制完整 JRE（含 `java.exe`/`javaw.exe`）
- 安全剪裁：移除 JavaFX（51MB）、Nashorn JS 引擎（2MB）、部署工具（3MB），197MB → 117MB
- 用 VBS 脚本调用捆绑的 `jre\bin\javaw.exe -jar` 启动应用
- 最终 ZIP 压缩后 **47MB**，解压后双击 `启动.vbs` 即可运行

**涉及文件**：
| 文件 | 动作 | 说明 |
|------|------|------|
| `dist/standalone/启动.vbs` | 新增 | VBS 启动脚本，调用捆绑 JRE 的 javaw |
| `dist/standalone/jre/` | 新增 | 剪裁后的 JRE（117MB，已 gitignore） |
| `NCHU-Image-Compressor-免安装版.zip` | 更新 | 47MB，解压即用 |

**教训**：
1. javapackager 的 `-native image` 模式创建的 EXE 并不可靠，它在不同 Windows 版本上的 `$APPDIR` 解析行为不一致
2. 最简单的方案往往最可靠：JRE + 启动脚本的组合比任何原生 EXE 包装器都稳定
3. 测试必须在目标使用场景下进行 — 命令行 OK ≠ 双击 OK
