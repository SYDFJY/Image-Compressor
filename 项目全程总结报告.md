## 一、项目概述

### 1.1 项目背景

在日常学习与工作中，图片与视频文件体积过大导致传输慢、存储占用多、网页加载延迟等问题频繁出现。市面上的在线压缩工具存在**隐私泄露风险**（文件上传至第三方服务器），专业软件（如 Photoshop、Premiere）**操作门槛高、授权费用昂贵**。本项目开发一款**离线、轻量、易用**的桌面图片/视频双模压缩工具，定位为"人人都能用的压缩工具"，面向普通用户而非专业设计师。

### 1.2 项目目标

| 目标维度 | 具体描述 |
|---------|---------|
| **核心功能** | 支持 JPG/PNG/BMP/GIF/WebP/TIFF/ICO 等格式的批量图片压缩 + MP4/AVI/MOV/MKV 等 8 种视频压缩，含质量/尺寸/格式/帧率/音频/裁剪调节 |
| **易用性** | 拖拽导入、右键菜单、全局快捷键、参数记忆、图片/视频一键模式切换、3 步完成压缩 |
| **用户体验** | 启动画面、Toast 通知、实时进度反馈、原图/效果图/对比三种预览模式、13 套可切换主题、搜索筛选排序 |
| **可靠性** | 单文件失败不影响整体、中途可取消、异常全面容错、EDT 非阻塞设计 |
| **可交付** | 绿色免安装 fat JAR + 捆绑 JRE+FFmpeg+VLC 便携版（267 MB 解压即用）+ 启动脚本 |

### 1.3 技术栈

### 1.4 技术约束（绝对红线）

1. ❌ 禁止使用 JDK 8 以上版本 API（`var`、`String.repeat()`、模块化等）
2. ❌ 禁止使用 JavaFX 等替代 GUI 框架
3. ❌ 禁止破坏 MVC 分层架构
4. ❌ 禁止在 EDT（事件调度线程）上执行耗时操作
5. ❌ 禁止擅自修改既定的配色/布局/交互规范
6. ❌ 禁止提交半成品、编译不通过、功能未验证的代码

## 二、需求分析

### 2.1 功能性需求全景

本项目共定义 **9 大功能模块**，覆盖图片压缩、视频压缩、文件管理、预览比较、配置持久化、主题系统等完整场景：

#### F1 文件导入（6 项）
- 单个/多个文件导入（JFileChooser）
- 整个文件夹导入（可选递归扫描）
- 从文件管理器拖拽文件到窗口
- 自动过滤不支持的文件格式并提示
- 自动去重（同一文件不重复添加）
- 单次导入上限 500 文件，防止 OOM

#### F2 文件列表管理（6 项）
- 显示文件名、原始大小、尺寸/时长、格式，图片显示缩略图
- 选中行高亮 + 缩略图预览
- 右键菜单（预览/打开原图/打开文件夹/单独压缩/移除/清空）
- Delete 键移除选中文件
- 支持清空全部文件 + Ctrl+Z 撤销
- 搜索/筛选/排序（按文件名/大小/格式）

#### F3 图片压缩参数设置（7 项）
- 压缩质量滑块（0-100），实时显示百分比，支持拖动实时预览
- 快捷预设按钮（标准 80 / 高效 60 / 高清 90）
- 缩放模式：不缩放 / 按百分比 / 按最大尺寸
- 输出格式：保持原格式 / JPEG / PNG / BMP / WebP
- 文件命名规则：添加后缀 / 添加前缀 / 自定义文件名（图片/视频均支持）
- 自定义输出目录 + 覆盖原文件开关
- GIF 专属：最大颜色数控制（2-256 级色阶量化）

#### F4 图片压缩执行（6 项）
- 单张图片压缩（质量 + 缩放 + 格式转换 + EXIF 保留/移除）
- 批量图片压缩（多线程并行，2 线程池）
- 实时进度反馈（进度条 + 当前文件名 + 百分比）
- 支持中途取消（Thread.interrupt() 机制）
- 单文件失败不影响整体（异常隔离）
- 无效参数组合检测（质量 100 + 不缩放 + 原格式 → 弹确认）

#### F5 预览与比较（6 项）
- 选中文件显示原图预览
- 压缩后自动切换效果图预览
- 三模式分段控件（原图 / 效果图 / 对比 — 同步缩放）
- 底部数据对比浮层（原始大小 / 压缩后 / 压缩率 / 尺寸）
- EXIF 信息展示面板（相机型号/拍摄参数/GPS/时间）
- 棋盘格背景标识透明区域

#### F6 结果与通知（4 项）
- 压缩完成后弹出结果统计弹窗（成功数/失败数/总耗时/节省体积）
- Toast 通知（成功/错误/信息/警告 4 种类型，队列依次显示）
- 弹窗中可打开输出目录
- 失败文件列表标红 + 错误信息 Tooltip

#### F7 配置持久化（5 项）
- 自动保存压缩参数（质量/格式/命名/缩放/输出目录）
- 自动保存窗口状态（位置/大小/最大化）
- 自动保存主题选择
- 损坏配置文件自动备份 + 重建默认（向后兼容新旧版本枚举值）
- 500ms 防抖写入，避免频繁磁盘 I/O

#### F8 主题系统（4 项）
- **13 套预设主题**：
  - 原始 4 套：蓝调经典 / 暗紫 / IntelliJ / Darcula
  - 蓝韵系列 6 套：天蓝 / 海蓝 / 靛蓝 / 青蓝 / 灰蓝 / 雾蓝
  - 莫兰迪色系：多巴胺暖（蜜桃橙→暖橙渐变）
  - 双模式：亮色 / 暗色
- 主题切换弹出面板（色块预览 + 名称 + 副标题）
- 一键切换无需重启（`SwingUtilities.updateComponentTreeUI()`）
- 主题自动记忆持久化

#### F9 视频压缩（12 项）
- 图片/视频模式一键切换（双段分段控件，图标+文字同时可见）
- 视频文件导入（MP4/AVI/MOV/MKV/WebM/FLV/WMV/M4V）
- 视频元信息显示（时长/分辨率/帧率/编码/码率），基于 ffprobe JSON 解析
- CRF 质量控制（滑块 0-51 映射到 100-0 显示，越低越清晰）
- 3 套预设方案（微信小视频 CRF30+480p / B站投稿 CRF20+1080p+30fps / 存档原画 CRF15+原始）
- 分辨率缩放（原始/480p/720p/1080p/4K）
- 帧率调整（保持原始/24/30/60 fps）
- 音频处理（保留/移除，编码器 H.264→AAC / WebM→Opus）
- 时间裁剪（起始时间 + 持续时长，FFmpeg `-ss` `-t` 参数）
- 输出格式选择（MP4 H.264 / WebM VP9 / AVI / MOV / MKV）
- 批量视频压缩 + 多线程并行（2 线程池）+ 进度反馈 + 中途取消
- FFmpeg 自动检测（启动时自动搜索常见安装目录 + 捆绑版路径，不可用时降级提示）

## 三、架构设计

### 3.1 MVC 分层架构

```
                        App.java (入口)
              SwingUtilities.invokeLater()
           SplashScreen → Theme → MainFrame
                → Controller.init()
                            |
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
        View (Swing)    Controller       Model (POJO)
        MainFrame       MainController    ImageFileInfo
        FileListPanel   ImageController   VideoFileInfo
        PreviewPanel    VideoController   CompressConfig
        ParamPanel                        VideoCompressConfig
        VideoParamPanel                   CompressResult
            │               │               ▲
            │               ▼               │
            └─────────>  Service  <─────────┘
                        CompressService
                        VideoCompressService
                        BatchCompressService
                        FileManagerService
                        ConfigService
                            │
                            ▼
                          Util
                     ImageCompressUtil
                     VideoCompressUtil
                     FileUtil / ImageUtil
                     VideoUtil / ThemeUtil
```

**依赖方向**（箭头 = "依赖于"）：
- **Model 层**：零依赖，纯数据模型（POJO + 枚举）
- **Util 层**：仅依赖 Model + JDK 标准库，无状态工具方法
- **Service 层**：依赖 Model + Util + 外部库，核心业务逻辑
- **Controller 层**：依赖 Service + Model + View，事件调度中心
- **View 层**：依赖 Controller + Model，纯界面绘制

### 3.3 核心数据流

```
用户拖拽文件
  → MainController.onDragImport()
  → FileManagerService.importFiles() [后台线程]
  → SwingUtilities.invokeLater()
  → FileListPanel.setFileList() [EDT]
  → 更新统计、启用压缩按钮

用户点击"开始压缩"
  → MainController.onStartCompress() [EDT]
  → 根据当前模式构建 CompressConfig / VideoCompressConfig
  → CompressWorker.execute() [SwingWorker 后台线程]
  → BatchCompressService.compressBatch()
    → [图片] CompressService → ImageCompressUtil → Thumbnailator
    → [视频] VideoCompressService → VideoCompressUtil → FFmpeg ProcessBuilder
  → worker.publish(ProgressChunk) → process() [EDT 更新进度条]
  → worker.done() → onCompressComplete() [EDT]
  → ResultDialog + 预览更新 + Toast 通知
```

### 3.4 线程安全设计

| 场景 | 策略 | 说明 |
|------|------|------|
| 耗时操作 | `SwingWorker.doInBackground()` | 文件导入/压缩执行在后台线程 |
| UI 更新 | `SwingUtilities.invokeLater()` | 所有 Swing 组件操作回到 EDT |
| 进度报告 | `SwingWorker.publish()/process()` | 自动在 EDT 上批量处理进度块 |
| 批量压缩 | `ExecutorService` 线程池 | 图片/视频各 2 线程并行 |
| 配置文件 | 500ms Timer 防抖 | 滑块频繁拖动不会触发多次 I/O |
| FFmpeg 子进程 | `Process.waitFor()` + 超时 | 1 小时超时保护 |
| 取消操作 | `Thread.interrupt()` + `destroyForcibly()` | 双层保护确保线程和子进程均释放 |

## 五、关键技术难点与解决方案

### 5.1 Thumbnailator 不调用 scale/size 导致静默失败 ✅

**现象**：格式转换时抛出 `"size is not set"` 异常，仅 PNG→JPEG（经 alphaFill 路径）正常。

**原因**：Thumbnailator 0.4.20 的 `Builder` 在调用 `toFile()` 前必须至少调用一次 `scale()` 或 `size()` 方法。`ScaleMode.NONE` 的原始实现为空操作（no-op），不满足此约束。

**解决**：`ImageCompressUtil.applyScale()` 的 NONE/default 分支改为 `builder.scale(1.0)` 显式保持原尺寸。

**教训**：第三方库的"默认行为"不可靠，关键路径必须有测试覆盖。此后每个 ScaleMode 分支都有明确的 `scale()` 调用。

### 5.2 Windows 文件锁定导致二次读取失败 ✅

**现象**：`ImageIO.read()` 读取尺寸后，`Thumbnails.of()` 再次打开同一文件时抛出 `IOException`。

**原因**：Windows 文件锁比 Linux 严格。`ImageIO.read()` 全量解码大图（~72MB）后可能未释放文件句柄（取决于 ImageReader 实现），导致 `Thumbnails.of()` 后续打开时遇到共享冲突。

**解决**：用 `ImageReader.setInput()` 仅读取 SOF（Start of Frame）标记段获取宽高（~2KB），替代 `ImageIO.read()` 全量解码。SOF 标记位于 JPEG 文件头部，不涉及色彩空间解码。此方案同时解决了 CMYK JPEG 解码失败问题。

**教训**：跨平台开发时必须注意文件锁定语义差异。Windows 默认以排他方式打开文件，Linux 则允许并发读取。

### 5.3 CMYK 色彩空间 JPEG 无法读取 ✅

**现象**：单反相机直出照片导入后尺寸显示 0×0，压缩失败。

**原因**：JDK 8 的 JPEG ImageReader 不支持 CMYK→RGB 色彩空间转换，`ImageIO.read()` 遇到 CMYK JPEG 时返回 null。这类文件通常来自专业相机（Adobe RGB / CMYK 色彩空间）或 Photoshop 导出。

**解决**：改用 `ImageReader.setInput()` 读取 SOF 标记段（仅含宽高数据，不涉及色彩空间解码），并在尺寸为 0 时给出明确提示："可能是 CMYK 色彩空间 JPEG，建议用画图/Photoshop 另存为标准 RGB 格式"。

**教训**：JPEG 格式的"标准"有多重变体（Baseline/Progressive/CMYK），不能假设所有 JPEG 都是 RGB 编码。生产级应用需引入 TwelveMonkeys 扩展插件覆盖更多色彩空间。

### 5.4 JDK 8 不支持 WebP 格式 ✅

**现象**：选择 WebP 输出格式后压缩静默失败。

**原因**：WebP 由 Google 在 2010 年发布，JDK 8（2014 年）的 ImageIO 无 Reader/Writer。JDK 15+ 才原生支持 WebP。

**解决**：**第一阶段**：从 6 个文件（枚举/下拉框/过滤器/配置恢复/文件工具/压缩工具）中联动移除所有 WebP 引用。**第二阶段**（v3.0）：引入 TwelveMonkeys ImageIO 3.10.1（最后支持 JDK 8 的版本），恢复 WebP 读写支持。

**教训**：引入新格式前必须验证 JDK 8 兼容性（Reader + Writer 双端检查）。JDK 8 ImageIO 原生仅支持 JPG/PNG/BMP/GIF/WBMP 五种格式的读写。

### 5.5 EDT 阻塞导致启动画面卡死 ✅

**现象**：蓝色 SplashScreen 弹窗无法关闭，只能强制结束进程。

**原因**：`App.java` 中 `Thread.sleep(300)` 在 EDT（事件调度线程）上执行。Swing 是单线程模型，sleep 期间无法响应任何 UI 事件（包括窗口关闭）。异常未捕获时 `dispose()` 永远不执行，JWindow 常驻屏幕。

**解决**：三层防护机制：
1. **Timer 替代 sleep**：`javax.swing.Timer` 在 EDT 上注册回调但不阻塞，300ms 后自动 `dispose()`
2. **try-finally 保护**：任何异常都会在 finally 中 `splash.dispose()`
3. **点击关闭兜底**：SplashScreen 添加 `MouseListener`，点击即可手动关闭

**教训**：EDT 是 Swing 的生命线，任何 `sleep()` / `wait()` / 同步 I/O 都不应出现在 EDT 上。

### 5.6 FlatLaf 主题切换后自定义组件颜色不更新 ✅

**现象**：`SwingUtilities.updateComponentTreeUI()` 后，自定义渲染的组件（分段控件、数据浮层）文字颜色不变。

**原因**：`updateComponentTreeUI()` 仅更新 LookAndFeel 管理的组件，不触发自定义 `paintComponent()` 的重新计算。自定义组件中直接写死了颜色常量而非从 UIManager 动态获取。

**解决**：建立集中的 **ThemeUtil 动态颜色注册系统**——将 UI 颜色通过 `UIManager.put()` 注册为 FlatLaf 属性，所有自定义组件通过 `UIManager.getColor()` 动态获取而非硬编码。主题切换时自动更新所有注册属性的值。

**教训**：自定义渲染组件必须与 LookAndFeel 系统耦合（通过 UIManager 代理），确保主题切换时所有颜色定义一处更新、全局同步。

### 5.7 FFmpeg 不在系统 PATH 导致视频元数据全显示「未知」 ✅

**现象**：视频导入后时长/分辨率/编码全部显示「未知」，即使 FFmpeg 已安装。

**原因**：`VideoUtil` 仅查找系统 PATH 中的 `ffprobe`，未考虑用户安装到自定义路径的情况。

**解决**：实现 **FFmpeg 自动发现**机制——按优先级搜索：
1. JVM 属性 `-Dffmpeg.bin.path`（捆绑版）
2. 系统 PATH 环境变量
3. 常见安装目录（`C:\ffmpeg\bin`、`C:\Program Files\ffmpeg\bin` 等）
4. 项目内置 `ffmpeg/bin` 目录

任一层找到即停止搜索，全部未找到时视频模式显示安装引导。

**教训**：外部依赖的路径查找应实现多层降级策略，不能只依赖单一机制。

### 5.8 视频缩放后编码失败（奇数宽高） ✅

**现象**：缩放视频（如竖屏 1080×1920 → 720p）后 FFmpeg 编码报错。

**原因**：H.264/VP9 的 yuv420p 像素格式要求宽高均为偶数。`scale=720:-1` 按比例计算高度可能得到奇数（如 `720:1280` → `720:1279` 整除问题），导致编码器拒绝。

**解决**：缩放滤镜改用 `scale=W:-2` 占位符——`-2` 表示自动计算该维度、保持宽高比、同时强制结果为偶数。`force_original_aspect_ratio` 配合确保不拉伸变形。

**教训**：视频编码对尺寸有严格约束（偶数宽高、最小尺寸、对齐要求），缩放计算必须考虑编解码器限制。

### 5.9 配置文件向后兼容（枚举版本迁移） ✅

**现象**：多主题系统升级后，旧 `config.json` 中 `"theme": "LIGHT"` 无法反序列化，程序崩溃。

**原因**：旧版 Theme 枚举仅含 `LIGHT`/`DARK`/`INTELLIJ`/`DARCULA`。升级到 13 套主题后移除了旧值，Gson 调用 `Theme.valueOf("LIGHT")` 抛出 `IllegalArgumentException`。

**解决**：注册自定义 `Gson TypeAdapter`，通过 `Theme.fromName()` 做新旧映射（LIGHT→BLUE_CLASSIC / DARK→DARK_PURPLE），外加 `getTheme()` null 兜底返回默认主题。

**教训**：配置持久化必须考虑向前兼容——旧版本配置文件在新版本中不能导致崩溃。枚举序列化应使用自定义名称映射而非依赖 `valueOf()`。

### 5.10 ffplay stdout 管道死锁 ✅

**现象**：使用 ffplay 播放视频时，界面卡死无响应，视频画面不出现。

**原因**：`ProcessBuilder` 启动 ffplay 后，Java 端同时读取 stdout 和 stderr 管道。ffplay 的 stdout 输出量远超管道缓冲区（默认 64KB），当缓冲区满时 ffplay 阻塞在 write()，Java 端若先读取 stderr（阻塞等待），则形成死锁——Java 等 stderr，ffplay 等 stdout 缓冲区排空，互不相让。

**解决**：将 stdout 排空放入独立线程（`FFmpeg-Stdout-Drainer` daemon thread），主线程只读取 stderr 解析进度。即使 stderr 暂时无输出，stdout 持续排空避免缓冲区满。

**教训**：`ProcessBuilder` 的输入/输出流必须全部消费，且不能以不确定的顺序等待。标准做法是 fork 独立线程消费每个流。
